Présentation

Le concept

Transformer un tirage au sort ordinaire en événement qui anime toute la salle.

Le problème

Aujourd'hui dans les casinos, un tirage au sort se résume a un animateur prennant un micro et annoncant une machine gagnante, Le moment manque de théâtralité — les joueurs ne regardent pas, l'émotion collective est absente.

Creative Aura change ça. Chaque machine à sous est équipée d'une light node discrète. Au moment du tirage, les lumières s'allument une par une, créent un effet de roulette dans toute la salle, ralentissent progressivement... jusqu'à ce qu'une machine s'illumine en or.

Ce que ça apporte
🎰
Expérience immersive
La lumière attire l'attention de toute la salle. Même les joueurs non participants regardent.
📱
Contrôle mobile
Le responsable lance le tirage depuis son téléphone en une seconde, sans ordinateur dédié.
⚡
Non intrusif
Bande LED sous la machine. Aucune modification de la machine elle-même. Pas d'homologation nécessaire.
🎨
Personnalisable
Couleurs, ambiances, nombre de gagnants — tout se configure depuis l'interface.
Les deux composants
🎛
Contrôleur — 1 par salle
Crée le réseau WiFi · Héberge le backoffice · Tire le gagnant · Envoie les commandes
💡
Light Node — 1 par machine
Se connecte au WiFi · Reçoit les commandes · Pilote la bande LED · Envoie un ping toutes les 5s
Le déroulement d'un tirage
1
Ouverture du backoffice
Connecté au WiFi CASINO_CTRL, le responsable accède à l'interface via 192.168.4.1 dans son navigateur.
2
Choix du mode et lancement
Roulette classique, multi-gagnantes ou plage personnalisée. Un seul bouton suffit.
3
Animation dans toute la salle
Flash blanc de départ, roulette qui tourne sur les machines, ralentissement progressif, suspense sur les dernières machines.
4
La machine gagnante s'illumine
Flashs lumineux, puis pulsation douce infinie. En multi-gagnantes, chaque gagnante garde sa couleur unique.
5
Célébration
En mode multi-gagnantes, toutes les machines gagnantes s'animent en arc-en-ciel pendant 3 secondes.

Modes de tirage

Trois façons d'animer votre salle selon l'événement.

Mode 1 — Roulette classique

La roulette tourne sur toutes les machines, une seule gagnante. Animation or avec pulsation douce infinie jusqu'au prochain tirage. Le mode idéal pour les soirées simples.

💡
Paramètre cléLe résultat est décidé AVANT le début de l'animation — exactement comme une vraie machine à sous. L'animation est une représentation théâtrale d'un résultat déjà calculé.
Mode 2 — Multi-gagnantes

De 2 à 6 gagnantes successives. Chaque roulette désigne une machine qui s'illumine dans une couleur unique et garde sa couleur pendant que les suivantes se jouent. À la fin, toutes les gagnantes célèbrent ensemble en arc-en-ciel.

Ambiances lumineuses

Votre salle vit au rythme de vos événements entre les tirages.

Entre les tirages, les machines ne sont pas éteintes. Le système propose 6 modes d'ambiance pour maintenir une atmosphère lumineuse cohérente avec chaque événement.

🌙
Veille
Bleu nuit très discret (20, 20, 40). Les machines restent présentes sans s'imposer.
💗
St Valentin
Fondu doux rose (220,20,100) vers rouge (200,0,0). Parfait pour les soirées romantiques.
🎄
Noël
Fond rouge avec scintillements blancs aléatoires. Ambiance fête de fin d'année.
🎃
Halloween
Fondu violet (148,0,211) vers orange (255,50,0). Une ambiance qui colle à la nuit.
🌈
Arc-en-ciel
Spectre complet HSV réparti sur les bandes. Festif et coloré.
🎨
Couleur libre
Choisissez n'importe quelle couleur depuis le sélecteur. Adaptable à votre charte.
Ambiances par zone

Chaque zone peut avoir sa propre ambiance indépendante. La commande NODE_AMBIANCE cible précisément les nodes d'une zone sans affecter les autres.

Chaque machine peut également recevoir une couleur individuelle indépendante.

Architecture globale

Système embarqué distribué — deux types de microcontrôleurs ESP32 communiquant en WiFi UDP broadcast sur un réseau dédié isolé.

Vue d'ensemble

Casino Light Node repose sur une architecture maître / esclaves entièrement sans fil. Un unique contrôleur ESP32 crée le réseau WiFi, héberge le backoffice web et orchestre tous les Light Nodes. Chaque Light Node est un ESP32 indépendant qui pilote sa bande LED WS2812 via la bibliothèque FastLED.

Documentation technique

Protocole de communication UDP

Protocole texte simple sur UDP — pas de confirmation, pas d'accusé de réception. Conçu pour la faible latence et la simplicité d'implémentation sur microcontrôleur.

Choix de l'UDP
✅ Pourquoi UDP
Latence minimale (~1ms sur LAN), broadcast natif (1 paquet → N nodes), pas de gestion de connexion, implémentation simple sur ESP32. La perte de paquets est négligeable sur un réseau local à courte portée (<1%).
❌ Pourquoi pas TCP/MQTT
TCP nécessite une connexion par node (overhead). MQTT nécessite un broker supplémentaire. Les deux ajoutent de la latence et de la complexité sans apporter de valeur pour ce cas d'usage.
Format des messages

Tous les messages sont des chaînes ASCII terminées sans délimiteur. Taille max du buffer de réception : 64 octets. Les commandes sont insensibles à la casse pour les numéros mais sensibles pour les mots-clés.

// Réception côté node
char msg[64];
memset(msg, 0, 64);
udp.read(msg, 63);
traiterCommande(String(msg));

Documentation technique

Backoffice web

Single Page Application hébergée directement sur l'ESP32 contrôleur. Zéro dépendance externe — HTML/CSS/JS vanilla stocké en flash PROGMEM.

Architecture frontend
Stockage
L'intégralité du HTML (environ 1500 lignes) est stockée dans un tableau const char INDEX_HTML[] PROGMEM en mémoire flash de l'ESP32. Chargée une seule fois au démarrage, elle ne consomme pas de RAM.
Polling
Le backoffice se rafraîchit via setInterval(ls, 5000) — requête HTTP GET /status toutes les 5 secondes. Pendant un tirage multi-gagnantes, le polling passe à 500ms pour détecter les nouvelles gagnantes en temps réel.

Optimisations UI — Menus stables

Le polling toutes les 5 secondes reconstruirait normalement tout le HTML et fermerait les menus ouverts. Deux fonctions silencieuses ont été implémentées pour éviter ça :

// Si un menu node est ouvert → mise à jour silencieuse des badges uniquement
function lnSilent() {
  fetch('/nodes').then(r => r.json()).then(d => {
    d.forEach(n => {
      var ns = document.getElementById('ns' + n.num);
      if (ns) { ns.className = 'ns ' + (n.online ? 'on2' : 'off2'); }
    });
  });
}

// Si aucun menu ouvert → reconstruction complète du HTML
function ln() { /* reconstruction complète */ }

// Sélection automatique dans le setInterval
setInterval(function() {
  var anyOpen = document.querySelector('.nr-body.open');
  if (anyOpen) lnSilent(); else ln();
}, 5000);

Même logique pour les zones avec lzSilent().

Animation roulette côté client

Le résultat du tirage est connu dès le retour de /lancer. L'animation JavaScript simule la roulette côté client en calculant les délais à l'avance :

function bs(nb, w, v, pmin) {
  // Phase rapide : nt tours × nb machines
  var nt = v >= 7 ? 2 : 1;
  // Ralentissement : 12 steps, easing cubique t³
  for (var i = 0; i < 12; i++) {
    var t = (i+1)/12, e = t*t*t;
    delai = Math.round(md + e * (380 - md));
  }
  // Phase finale : jusqu'à 8 machines avant la gagnante
  // délai : 380ms → 500ms progressivement
}

Les délais JS sont synchronisés avec les délais UDP côté contrôleur — les deux utilisent les mêmes formules pour garantir la cohérence entre l'affichage mobile et les LEDs physiques.

Animations LED

Toutes les animations sont non-bloquantes — gérées par machine à états dans la boucle principale via millis(). Aucun delay() n'est utilisé dans loop().

Principe non-bloquant
// loop() sur le node — s'exécute ~1000x/seconde
void loop() {
  gererAnimation();    // flash gagnant
  gererPulsation();    // pulsation gagnant
  gererCelebration();  // arc-en-ciel 3s
  gererFondu();        // ambiances oscillantes
  gererNoel();         // scintillements

  // Réception UDP (non-bloquant)
  int taille = udp.parsePacket();
  if (taille) { /* traitement */ }

  // Ping toutes les 5s
  if (millis() - dernierPing > 5000) { /* envoi ping */ }
}
ℹ️
Chaque fonction de gestion vérifie son propre timestamp via millis() et retourne immédiatement si ce n'est pas encore le moment d'agir. Cela garantit que le node reste réactif aux commandes UDP même pendant une animation longue.

Phase de ralentissement roulette — Détail
// Contrôleur — gererTirage()
int vitesseBase = 18 + (10 - vitesse) * 4; // ex: vitesse=7 → 30ms

// Phase rapide : 2×N steps (si vitesse≥7) ou 1×N steps
int toursRapides = (vitesse >= 7 ? 2 : 1) * nbPlage;

// Ralentissement : 12 steps, easing puissance 3
for step in 0..12:
  t = step / 12.0
  eased = t³
  delai = vitesseBase + eased × (380 - vitesseBase)
  // vitesse=7: de 30ms → 380ms en 12 steps non-linéaires

// Phase finale : suspense progressif
stepsVersGagnant = (gagnant - position + N) % N
if stepsVersGagnant > 8 → delai = 380ms
else → delai = 380 + (8-steps+1)/8 × 120ms  // jusqu'à 500ms

Configuration des Light Nodes

Procédure de premier démarrage et remise en configuration — utilise un serveur web embarqué temporaire sur le node lui-même.

Logique de démarrage
void setup() {
  // 1. Init FastLED
  FastLED.addLeds<WS2812, LED_PIN, GRB>(leds, NUM_LEDS);
  FastLED.setBrightness(80);  // 31% du max

  // 2. Lecture mémoire flash
  prefs.begin("node", true);  // read-only
  monNumero = prefs.getInt("numero", 0);  // 0 si jamais configuré
  prefs.end();

  // 3. Branchement selon état
  if (monNumero == 0) {
    modeConfiguration();  // LEDs orange + AP WiFi NODE_XXXXXX
  } else {
    modeNormal();  // Connexion CASINO_CTRL + LEDs vertes → veille
  }
}
Mode configuration — Détail technique
void modeConfiguration() {
  // Suffixe MAC unique : 3 derniers octets de l'adresse WiFi
  apName = "NODE_" + getMacSuffix();  // ex: NODE_A3F2C1

  WiFi.softAP(apName.c_str(), "config123");
  // → IP du node en mode config : 192.168.4.1

  // LEDs orange = signal visuel de l'état
  for (int i = 0; i < NUM_LEDS; i++) leds[i] = CRGB::Orange;
  FastLED.show();

  // Serveur web minimal pour la configuration
  server.on("/", handleRoot);          // Formulaire HTML
  server.on("/save", HTTP_POST, handleSave);  // Sauvegarde + reboot
  server.begin();

  while (true) { server.handleClient(); }  // Boucle bloquante
}

Procédure iPhone obligatoireEntre chaque node, aller dans Réglages → WiFi → (i) → Oublier ce réseau. Sans cette étape, l'iPhone réutilise le cache du réseau précédent et peut attribuer le même numéro à deux nodes.

Évolutions prévues

Roadmap technique organisée par horizon temporel. Voir le document Roadmap complet pour le détail de chaque item.

Court terme — 0 à 3 mois
  • Tests stabilité 10-20-40 nodes — validation en conditions réelles avec charges WiFi croissantes
  • Authentification backoffice — Basic Auth ou formulaire HTML sur ESP32, niveaux opérateur / superviseur
  • BRIGHTNESS configurable depuis backoffice — commande UDP BRIGHTNESS:X + FastLED.setBrightness(X) + sauvegarde flash
  • NUM_LEDS configurable sans recompilation — saisie lors de la configuration du node, stockée en Preferences
  • Historique tirages persistant — 100 derniers tirages en flash NVS, accessible via GET /history
Moyen terme — 3 à 12 mois
  • Mise à jour OTA (Over The Air) — firmware .bin uploadé via backoffice, distribué aux nodes via ArduinoOTA
  • Mode démo autonome — tirages automatiques configurable (intervalle, mode, vitesse) sans intervention opérateur
  • Support 100+ nodes — routeur WiFi dédié (ESP32 en mode STA), zéro modification firmware node
  • Raspberry Pi contrôleur — backoffice Node.js, 100+ connexions WiFi, stockage persistant natif
Long terme — 12+ mois
  • Version RS485 filaire — bus RS485 jusqu'à 1200m, module MAX485 par node, protocole identique
  • Certification GLI / BMM — True RNG hardware, journal cryptographique, audit tiers
  • API REST formelle — versioning v1, token, documentation OpenAPI/Swagger
